Three steps. Done.

Your artifact belongs next to the diff.

Stop dropping screenshots into temporary chats, personal gists, or mystery buckets. One command validates the file, chooses a viable upload path, and puts the result where review happens.

  1. Point at the evidence

    Images, GIFs, and videos. Add one file or a whole before-and-after set.

    ./shot.png ./after.gif

  2. Run one command

    Target a PR directly, or let pr-media resolve the open PR for your current branch.

    pr-media add ./shot.png --pr 42

  3. Keep review moving

    The rendered media lands in a comment or the PR description, ready for the reviewer.

Choose a route, not an upload hack

Media needs the same care as code.

A review artifact can be private, public, or native to GitHub’s composer. pr-media makes the trade-offs explicit; auto selects from the viable routes for the environment.

One command, three deliberate paths. Keep a browser session inside the browser, use repo-scoped Git data in private CI, or use GitHub Releases when their repository-level access model is the right fit.

Cookie replay

Convenient native uploads can carry a full-account browser credential.

pr-media drives the active browser UI when needed; it does not read or serialize the cookie that authenticates it.

Release storage

Release assets follow the repository’s access model.

They are public in public repositories and require repository access in private ones; hidden refs are a separate, repo-scoped route for review evidence.

Automation

CI and coding agents need a contract, not a browser-only workaround.

--dry-run, --json, fallback in auto mode, and cleanup make media a dependable review step.

Three upload paths

Use what the environment can support.

Local browser available? Use the same attachment flow as GitHub’s web UI. Running headless? Fall back to documented GitHub APIs through gh.

01

browser

Drives your already-authenticated browser, reads the URL GitHub inserts, clears the draft, and never clicks Comment.

user-attachmentsBest on your laptop
02

hidden-ref

Stores the file on a dedicated Git ref through the Git Data API. Private-repository URLs continue to require repository access.

gh apiBest for private CI
03

release

Publishes the file as an asset on a dedicated prerelease. Public repositories produce public asset URLs; private repositories require repository access.

gh releaseBest for public CI
Private repository?

Prefer hidden-ref or browser. Treat every media artifact like code: its visibility should match the review context, and viewers still need the relevant GitHub access.

Made for agents, too

Give the coding agent a proof step.

A bundled skill teaches an agent when to use pr-media, while --json gives automation stable, parseable output.

  • Plan safely. --dry-run resolves the PR and validates files without uploading.
  • Parse reliably. --json returns the name, URL, Markdown, and chosen strategy.
  • Review visually. The agent can put generated UI evidence directly next to its code changes.

A narrow security promise

Your browser session stays in the browser.

pr-media never extracts or replays your GitHub session cookie. Headless strategies use the credentials already managed by the gh CLI.

No cookie-store access

The browser strategy uses the active UI. It does not read a browser cookie database or serialize credentials.

gh-managed authentication

API calls run through the GitHub CLI. pr-media does not write a token to disk or print one in its output.

Arguments, not shell strings

External commands receive an argument array, reducing command-injection risk from file paths, repository names, and PR URLs.
